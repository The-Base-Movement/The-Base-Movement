// Web Dev Supervisor Protocol

Rules of
Engagement

A no-nonsense operating system for building, shipping, and owning web products. Move fast. Think first. Break nothing critical.

01 // Core Aims
01
🎯

Ship Working Product

A deployed imperfect product beats a perfect repo that never goes live. Set weekly ship targets. Nothing ships without a working URL.

  • Define "done" before you start building
  • Every feature must have a clear user value
  • Kill features that delay core launch
02
⚡

Ruthless Simplicity

Every line of code is a liability. Every dependency is a risk. Remove before you add. Complexity is the enemy of speed.

  • Question every npm package install
  • Prefer native browser APIs over libraries
  • No abstraction without 3 use cases to justify it
03
📐

Performance Is a Feature

Sub-2s load time is a non-negotiable target. Users don't care about your architecture; they care about speed.

  • Lighthouse score ≥ 90 before any launch
  • Core Web Vitals in green on every PR
  • Image optimization is mandatory, not optional
04
💰

Revenue-Linked Goals

Technical decisions must map to business outcomes. Every sprint goal must answer: how does this make or save money?

  • Track conversions, not just traffic
  • Monetization layer planned before build starts
  • Hosting/infra costs reviewed monthly
05
🔁

Iterate Fast, Break Nothing Critical

Speed without stability is chaos. Rapid iteration is the goal — but core flows (auth, payment, data) need full regression before each push.

  • Feature flags for risky rollouts
  • Rollback plan exists before every deploy
  • Staging environment mirrors production
06
🧠

Own the Decision

No hiding behind "it's just how the framework works." You chose the stack. You own the consequences. Document architectural decisions.

  • ADRs written for every major tech choice
  • No "I assumed" — verify and confirm
  • Post-mortems within 48hrs of any failure
02 // The 6 Iron Laws
I

No Deploy on Friday After 3PM

Production changes on Friday afternoons are how you ruin weekends. Emergency hotfixes are the only exception. If it can wait until Monday, it waits.

II

Every Commit Must Be Reversible

Use feature branches. Squash before merge. Tag releases. If you can't revert a change in under 5 minutes, you haven't engineered it correctly.

III

No Secret Config in Codebase

API keys, DB credentials, and tokens live in environment variables only. Every .env file is in .gitignore from day zero. No exceptions, ever.

IV

Review Before Merge — Always

No self-merging to main/production. At minimum, a 10-minute self-review against the checklist. On team projects, a second set of eyes is non-negotiable.

V

If It's Not Monitored, It Doesn't Exist

Uptime monitoring, error tracking (Sentry or equivalent), and real user metrics must be live before any project is considered "launched." Blind deployments are amateur hour.

VI

Eliminate, Then Automate, Then Optimize

Don't automate a broken process. First ask: does this step need to exist? If yes, automate it. Only then optimize. This is the Musk algorithm applied to web dev.

03 // Security Measures
Severity Area Rule Enforcement
CRITICAL Auth & Sessions Never roll your own auth. Use proven providers (NextAuth, Clerk, Supabase Auth). HTTPOnly cookies, CSRF tokens mandatory. Code review block
CRITICAL SQL / DB Parameterized queries only. Zero raw string interpolation in SQL. ORM used or manual escaping reviewed. Automated lint rule
CRITICAL Secrets Secrets in .env only. git-secrets or trufflehog scan on every commit. Rotate on any suspected exposure immediately. Pre-commit hook
HIGH Input Validation All user inputs validated server-side. Zod / Yup schemas on every form and API endpoint. Never trust client data. PR checklist item
HIGH HTTPS / Headers HTTPS everywhere. Set: CSP, X-Frame-Options, HSTS, X-Content-Type-Options. Use helmet.js or equivalent framework defaults. Deployment check
HIGH Dependencies Run `npm audit` before every deploy. No packages with critical vulnerabilities ship. Dependabot or Renovate enabled on all repos. CI/CD gate
STANDARD Rate Limiting Rate limiting on all API routes. Auth endpoints get stricter limits. Use Upstash Redis or equivalent for distributed rate limiting. Code review
STANDARD File Uploads Validate file type server-side (not by extension). Enforce size limits. Store outside webroot. Never serve user files with execute permissions. Feature review
STANDARD Error Handling Never expose stack traces to users. Log errors internally (Sentry). Return generic user-facing messages. No server info in response headers. Code review

05 // Priority Framework
P0 — Critical

Fix Now. Nothing Else.

Site down, data breach, broken auth, payment failure. Drop everything. No estimates given — just fixed. Alert all stakeholders immediately.

P1 — High

Fix This Sprint

Major UX breakage, security vulnerability (non-critical), significant performance regression. Addressed within current sprint. No new features until resolved.

P2 — Standard

Scheduled Work

Feature builds, optimizations, minor bugs. Goes into the backlog. Prioritized by revenue impact and user value — not by what's fun to build.
